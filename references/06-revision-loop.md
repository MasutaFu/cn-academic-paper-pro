# 06. 迭代修订工作流

C 刊论文通常需要 5–30 轮迭代。本文件给出一套已经过 30+ 轮实战验证的修订方法论。

---

## 一、迭代节奏总览

```
v1–v3    搭骨架             框架、问题意识、初稿成型
v4–v8    结构性修复         理论挂旗、操作化、因果链
v9–v15   内容性修复         论证深度、文献对话、案例细节
v16–v25  语体打磨           去 AI 化、句式优化
v26+     投稿适配           摘要、格式、参考文献、英文摘要
```

**不要试图在一稿内完成所有维度**。结构未稳前去 AI 化是浪费工夫；初稿未成型前参考文献体系化也是浪费。

---

## 二、单轮修订的标准流程

### 2.1 触发源

通常是以下三种之一：

1. **自评轮**：用户说"看看这一稿还有什么问题"
2. **外审轮**：用户上传了审稿意见 docx
3. **专项轮**：用户指定要解决某类问题（如"去 AI 化""文献补强"）

### 2.2 标准步骤

```
步骤 1：读稿（用户的最新版本）
步骤 2：读补充材料（审稿意见 / 上一轮修改清单）
步骤 3：执行评估，按 P0/P1/P2 分层标注
步骤 4：输出"修改清单"（不动文件）
步骤 5：等用户确认 / 选择 / 修正
步骤 6：执行修改，输出新版 docx
步骤 7：附"修改总账"，列出本轮做了什么、悬而未决的是什么
步骤 8：给出"下轮建议"
```

**禁止跳步**。尤其禁止从步骤 3 直接跳到步骤 6——这样做的代价是用户拿到一个他不想要的版本，要重做。

## 三、P0/P1/P2 优先级分层

### P0：致命问题（不解决会被毙）

判定标准（满足任一即为 P0）：
- 概念错位（如理论术语用错）
- 涉台、政治措辞违规
- 参考文献缺号、引而未列、伪造文献
- 摘要与正文承诺断裂
- 操作化缺失（核心概念未落地）
- 因果链或论证链显著断裂
- 数据 / 事实错误（如"十五五"规划状态错误）
- 投稿期刊的硬性体例违反（摘要超字数、缺英文摘要等）

### P1：重大问题（解决后大幅提升录用率）

- 理论挂旗（引言罗列但正文不用）
- 文献对话薄弱（中文期刊文献过少）
- 案例分析深度不够（只描述不解释）
- 章节内部重复叙述
- 章节之间逻辑不衔接
- 表格信息密度低或与正文口径不一致
- 语体大面积 AI 化

### P2：建议改进（锦上添花）

- 局部语体打磨
- 个别句式优化
- 参考文献条目微调
- 图表美化
- 个别术语替换

### 输出格式

每个问题用以下模板：

```markdown
## [P0-1] 概念错位：三螺旋"边界条件"

**位置**：第二部分（三）末段（第 79 行）

**原文**：
> 这种排他性绑定反映的是三螺旋理论的边界条件，即企业与大学的双螺旋锁定而不是三方均衡互动。

**问题**：三螺旋理论中没有"边界条件"这一表述，属于概念硬伤。Etzkowitz 框架讨论的是"互动空间""混合组织""知识资本化"等概念。

**建议改为**：
> 这种排他性绑定偏离了三螺旋理论所设想的三方均衡互动模式，呈现为企业—大学双向锁定的简化形态。

**优先级理由**：熟悉 Etzkowitz 文献的审稿人会一眼挑出，是确定性扣分项。
```

## 四、修改清单的物料标准

### 4.1 必含字段

每条修改至少包含：
- 优先级标签（P0/P1/P2）
- 位置（段号、行号或上下文锚点）
- 原文片段
- 改后文本
- 修改理由

### 4.2 不允许的格式

- 只说"需要修改 X 部分"不给替换文本——必须给
- 只说"建议优化句式"不给具体方案——必须给
- 用"等等""诸如此类"代替具体清单——必须穷举

### 4.3 修改总数控制

单轮通常 5–15 条，超过 15 条说明问题没收敛，建议拆为"结构性修改轮"和"语体打磨轮"两批。

---

## 五、修改总账模板

每轮修改完成后输出：

```markdown
# vN → vN+1 修订总账

## 本轮处理（已完成）

### P0 修复（共 X 处）
1. ✅ [问题简述] — [改动简述]
2. ✅ ...

### P1 修复（共 Y 处）
1. ✅ ...

### P2 修复（共 Z 处）
1. ✅ ...

## 本轮调整（结构性）

- 章节合并 / 拆分：...
- 参考文献新增 / 删除：[N] 条
- 表格修改：...
- 字数变化：原 XXXX 字 → 现 XXXX 字

## 悬而未决（下轮处理）

- [ ] 项目 1：原因
- [ ] 项目 2：原因

## 下轮建议优先级

1. 优先处理 X，理由是 ...
2. 其次处理 Y，理由是 ...
```

## 六、外审意见处理流程

### 6.1 读审稿意见的姿态

不是照单全收，而是评估：

| 审稿建议类型 | 处理方式 |
|---|---|
| 概念 / 事实硬错 | 必改 |
| 结构性建议（章节调整、框架修正） | 严肃评估，多数采纳 |
| 文献补充建议 | 评估相关度，相关则采纳，无关则注明 |
| 语体 / 表达建议 | 评估是否会稀释学术品质，会则礼貌拒绝 |
| 与用户既定决议冲突的建议 | 列出冲突点，请用户裁决 |

### 6.2 回应稿格式

正式投稿回复需要：

```markdown
尊敬的审稿专家：

感谢您对本文的细致审阅。我们对您提出的 N 条意见逐一回应如下，正文修改之处已用修订模式标出。

## 意见 1：[审稿原文]

**回应**：[采纳 / 部分采纳 / 不采纳]

**修改内容**：[具体说明]，详见正文第 X 页第 Y 行。

[如不采纳]：经反复斟酌，我们认为……（给出学理理由），故未作修改，请审稿专家鉴谅。

...
```

### 6.3 多份审稿意见的统合

收到两份及以上审稿意见时：

1. 先列出"共识项"——两位都提到的问题，必改
2. 再列出"分歧项"——两位意见相反的，需用户裁决
3. 最后列出"单独项"——只一人提到，逐条评估

## 七、版本管理

### 7.1 文件命名规则

```
vNN-[标题关键词]-[状态标记].docx
```

例：
- `v33-发达经济体集成电路人才培养体系战略性重构的比较研究.docx`
- `v34-审稿意见1回应稿.docx`
- `v35-投稿定稿.docx`

### 7.2 关键节点保留

不删除中间版本。以下版本必须留存：
- 每个大轮次的起始版本（v1, v5, v10, v15, v20, v25, v30）
- 每次外审前的版本
- 投稿版本

### 7.3 修改清单归档

每轮的修改清单作为独立 .md 文件保留：
- `vNN_修改清单.md`
- `vNN_修订总账.md`
- `vNN_审稿意见N.docx`（如适用）

## 八、典型工作流示例

### 示例 1：用户上传新一稿，要求审阅

**正确流程**：
1. 读稿 → 通读全文一遍
2. 调用 `F-self-review.md` 的多专家审稿清单逐项扫描
3. 输出 P0/P1/P2 分层清单
4. 在末尾问"是否按这个清单执行？或者只处理 P0？"
5. 等用户回复

**错误流程**：
1. 读稿
2. 直接改 docx
3. 输出新版

### 示例 2：用户说"按你说的改"

**正确流程**：
1. 确认上一轮清单是否被全部接受（如果没明确表态，问一句）
2. 依次执行清单（可用脚本批处理）
3. 自检（运行 H-banned-words.md 的扫描脚本）
4. 输出 docx + 修订总账

**错误流程**：
1. 凭印象改
2. 没自检
3. 输出未确认是否改对的版本

### 示例 3：用户说"看看哪里还有 AI 痕迹"

**正确流程**：
1. 加载 `A-style-guide.md` 和 `H-banned-words.md`
2. 运行扫描脚本（grep 禁用词、句首禁语、密度词）
3. 逐段读，识别第三层问题（对称结构、单向论证、段尾升华）
4. 输出 P0/P1/P2 清单
5. 建议先处理第三层、再处理第一二层

## 九、迭代终止条件

什么时候认为可以投稿了？以下 8 项全过：

- [ ] 摘要符合期刊字数要求且与正文承诺一致
- [ ] 中英文摘要齐全
- [ ] 关键词数量符合期刊要求
- [ ] 参考文献无缺号、无引而未列、按 GB/T 7714-2015 顺序编号
- [ ] 涉台 / 政治措辞合规
- [ ] H-banned-words.md 扫描脚本全部通过
- [ ] 各章节小结无"综上所述"类规范性收口
- [ ] 结语含至少一处局限说明

未全过不投。
